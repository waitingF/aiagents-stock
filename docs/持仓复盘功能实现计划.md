# 持仓复盘功能实现计划

## 实现状态

当前分支已完成 V1 实现：

- 已自动迁移 `portfolio_analysis_history`，新增完整决策和复盘相关字段。
- 已保存 `operation_advice`、`holding_period`、`position_size`、`risk_warning`、`final_decision_json`。
- 已新增确定性复盘规则，基于上一条分析和本次分析判断“首次分析/兑现/部分兑现/失效/待验证”。
- 已保存 `review_status`、`review_summary`、`review_json`。
- 已在持仓历史 UI 展示操作建议、风险提示、持仓建议和复盘信息。
- 已在批量分析结果卡片展示本次复盘信息。
- 已在邮件/Webhook 通知中加入持仓复盘摘要。
- 已将复杂进场区间解析统一为“提取前两个价格数字”，兼容 AI 返回的多区间说明文本。

后续增强项：

- 引入分析周期内最高价/最低价，提升止盈、止损和进场区间触发判断的准确度。
- 在现有结构化复盘结果基础上增加可选 AI 润色，不影响批量分析主流程。
- 为复盘规则补充自动化单元测试文件，纳入 CI。

## 背景

当前“持仓分析”更接近一次性的批量分析记录：系统保存评级、目标价、进场区间、止盈止损、信心度和摘要，用于回看当时的分析结论。

但它还没有形成真正的“复盘”能力。用户在历史记录中无法直接看到：

- 上一次判断是否兑现。
- 买点、止盈、止损是否触发。
- 本次结论相比上次发生了什么变化。
- 原有持仓逻辑是否仍然成立。
- 接下来应该重点观察哪些风险或催化因素。

因此需要在现有持仓分析链路上增加“前后对比 + 复盘结论”。

## 当前问题

1. 持仓历史表 `portfolio_analysis_history` 保存的信息偏少，只保存结构化交易字段，没有完整保存 AI 最终决策。
2. `summary` 字段当前设计为分析摘要，但实际 AI 返回字段主要是 `operation_advice`，容易出现摘要为空或信息不完整。
3. 每次持仓分析之间彼此独立，没有读取上一条历史记录做对比。
4. UI 只展示“当时怎么看”，没有展示“上次判断对不对、本次怎么修正”。
5. 定时分析通知只汇总本次分析结果，没有突出复盘变化。

## 目标

新增股票复盘能力后，每次持仓分析应同时回答两个问题：

1. 本次怎么看这只股票。
2. 相比上次分析，判断是否兑现，逻辑是否变化，操作计划是否需要调整。

复盘结果需要结构化保存，并在持仓历史、定时报告和通知中可读展示。

## 功能范围

### 第一阶段：补齐当前历史信息

目标是让现有持仓历史记录更完整，不改变主流程。

- 保存摘要时兼容 `operation_advice`、`advice`、`summary`。
- 增加完整最终决策字段 `final_decision_json`。
- 保存持有周期、仓位建议、风险提示。
- UI 历史记录展示：
  - 操作建议。
  - 持有周期。
  - 仓位建议。
  - 风险提示。

建议字段：

```sql
ALTER TABLE portfolio_analysis_history ADD COLUMN operation_advice TEXT;
ALTER TABLE portfolio_analysis_history ADD COLUMN holding_period TEXT;
ALTER TABLE portfolio_analysis_history ADD COLUMN position_size TEXT;
ALTER TABLE portfolio_analysis_history ADD COLUMN risk_warning TEXT;
ALTER TABLE portfolio_analysis_history ADD COLUMN final_decision_json TEXT;
```

### 第二阶段：新增复盘结果

目标是在每次新分析保存前，读取上一条历史记录，生成复盘。

建议先直接扩展 `portfolio_analysis_history`：

```sql
ALTER TABLE portfolio_analysis_history ADD COLUMN review_status TEXT;
ALTER TABLE portfolio_analysis_history ADD COLUMN review_summary TEXT;
ALTER TABLE portfolio_analysis_history ADD COLUMN review_json TEXT;
```

`review_status` 建议枚举：

- `兑现`
- `部分兑现`
- `失效`
- `待验证`
- `首次分析`

`review_json` 建议结构：

```json
{
  "review_status": "部分兑现",
  "price_change_pct": 3.42,
  "target_hit": false,
  "entry_hit": true,
  "take_profit_hit": false,
  "stop_loss_hit": false,
  "rating_change": "持有 -> 买入",
  "plan_check": "上次建议等待回踩买入，本轮最低价已进入进场区间但未触发止损。",
  "logic_change": "资金面改善，技术面从弱修复转为突破确认。",
  "risk_change": "主要风险从破位风险转为追高回落风险。",
  "action_adjustment": "继续持有，若回踩不破关键支撑可小幅加仓。",
  "next_watchpoints": [
    "是否站稳20日均线",
    "主力资金是否连续流入",
    "是否跌破上一轮止损位"
  ]
}
```

### 第三阶段：复盘生成逻辑

新增复盘服务函数，例如：

```python
def generate_portfolio_review(
    previous_record: dict,
    current_result: dict,
    stock_info: dict,
    indicators: dict,
) -> dict:
    ...
```

处理流程：

1. 查询该股票上一条分析历史。
2. 若不存在上一条记录，返回 `review_status = "首次分析"`。
3. 根据上一条记录和当前价格做基础量化判断：
   - 当前价相对上次分析价涨跌幅。
   - 是否进入上次进场区间。
   - 是否达到上次目标价。
   - 是否触发上次止盈。
   - 是否触发上次止损。
   - 评级是否变化。
4. 将基础判断、上一条历史、本次最终决策交给 AI 生成复盘文本。
5. 将复盘 JSON 保存进本次分析历史。

基础判断可以先用当前价近似实现；后续可引入区间内最高价、最低价，提升“是否触发”的准确性。

### 第四阶段：AI Prompt

新增一个专门用于持仓复盘的 prompt，不建议混在 `final_decision` 里。

Prompt 需要输入：

- 股票代码和名称。
- 上次分析时间、评级、价格、目标价、进场区间、止盈、止损、操作建议。
- 本次价格、评级、目标价、进场区间、止盈、止损、操作建议。
- 基础触发判断结果。

输出要求：

```json
{
  "review_status": "兑现/部分兑现/失效/待验证/首次分析",
  "plan_check": "上次交易计划执行情况",
  "price_review": "价格表现复盘",
  "logic_change": "本次核心逻辑变化",
  "risk_change": "风险变化",
  "action_adjustment": "操作调整建议",
  "next_watchpoints": ["观察点1", "观察点2", "观察点3"],
  "review_summary": "100字以内复盘摘要"
}
```

### 第五阶段：UI 展示

在持仓历史卡片中新增“复盘信息”区域：

- 复盘状态：兑现/部分兑现/失效/待验证/首次分析。
- 上次计划执行情况。
- 价格表现复盘。
- 核心逻辑变化。
- 风险变化。
- 本次操作调整。
- 下次关注点。

建议布局：

- 第一行仍展示价格信息、进场区间、风控位置。
- 第二行展示“操作建议”和“复盘信息”。
- `review_status` 使用颜色标签：
  - 兑现：绿色。
  - 部分兑现：黄色。
  - 失效：红色。
  - 待验证：灰色。
  - 首次分析：蓝色。

### 第六阶段：定时分析报告和通知

定时持仓分析完成后，在通知内容里增加复盘摘要：

- 评级发生变化的股票。
- 触发止盈/止损的股票。
- 已进入上次进场区间的股票。
- 复盘状态为“失效”的股票。
- 本次建议需要调整仓位的股票。

通知示例：

```text
【持仓复盘摘要】
003816 中国广核：部分兑现，上次进场区间已触达，本次维持买入，关注是否站稳4.77。
603288 海天味业：待验证，仍未有效突破压力位，本次维持持有，跌破37.30需止损。
```

## 推荐落地顺序

1. 修复 `summary` 保存逻辑，兼容 `operation_advice`。
2. 扩展历史表字段，保存完整最终决策 JSON。
3. UI 展示更多已有最终决策字段。
4. 新增复盘生成函数，先实现非 AI 的基础触发判断。
5. 增加 AI 复盘 prompt，输出结构化 `review_json`。
6. UI 展示复盘信息。
7. 定时报告和通知增加复盘摘要。
8. 增加迁移兼容逻辑，确保老数据库自动补字段。

## 风险和注意事项

- 老数据库需要兼容迁移，`ALTER TABLE ADD COLUMN` 要检测字段是否已存在。
- 复盘判断如果只用当前价，无法准确判断分析周期内是否曾触达止盈止损；后续建议引入区间最高价和最低价。
- AI 输出必须强制 JSON，并对解析失败设置兜底结果。
- 复盘结果应保存原始 JSON，避免后续 UI 或通知需要字段时只能从文本里解析。
- 首次分析没有上一条记录，应明确展示“首次分析”，不要伪造复盘结论。

## 验收标准

- 新分析记录能保存完整最终决策和复盘 JSON。
- 首次分析显示“首次分析”。
- 第二次及后续分析能展示相对上一条记录的复盘信息。
- 评级变化、触发止盈、触发止损、进入进场区间能被识别。
- UI 历史记录能直接看到操作建议和复盘摘要。
- 定时通知包含复盘摘要。
- 老数据库升级后不影响现有持仓管理和历史查看。
