# 项目模块化改造方案

本文档用于跟进 `aiagents-stock` 从“顶层散文件 + Streamlit 单体入口”改造为“按业务域分包 + 清晰基础设施层”的过程。目标是在不改变现有用户功能的前提下，逐步降低 `app.py` 和顶层文件复杂度，让后续新增策略、数据源、智能体、报告和监控能力更容易维护。

## 1. 当前项目功能概览

项目当前是一个基于 Streamlit 的多智能体股票分析系统，主要能力如下：

### 1.1 单股与批量分析

- 支持 A 股、港股、美股基础信息和历史行情获取。
- 支持技术指标计算、财务数据、季报、资金流、风险事件、市场情绪、新闻数据采集。
- 通过多智能体完成技术面、基本面、资金面、风险、情绪、新闻等分析。
- 支持团队讨论、最终投资决策、历史记录保存、PDF/Markdown 报告导出。
- 支持批量股票顺序分析和多线程并行分析。

### 1.2 选股策略

- 主力选股：基于主力资金净流入、板块热点和 AI 综合筛选。
- 低价擒牛：低价高成长股票筛选与策略监控。
- 小市值策略：小盘高成长股票筛选。
- 净利增长策略：净利润增长稳健股票筛选。
- 低估值策略：低 PE、低 PB、高股息、低负债价值投资筛选。

### 1.3 策略研判

- 智策板块：市场、板块、概念、资金、情绪综合研判。
- 智瞰龙虎：龙虎榜数据、AI 评分、历史报告、批量分析。
- 新闻流量：热点新闻、流量趋势、情绪分析、预警。
- 宏观分析：国家统计局宏观数据、行业映射、优质标的筛选。
- 宏观周期：康波周期、美林投资时钟、中国政策分析。

### 1.4 投资管理与监控

- 持仓分析：组合持仓、定时分析、历史记录。
- 实时监测：价格监控、通知提醒、定时调度。
- AI 盯盘：DeepSeek AI 决策、K 线展示、MiniQMT/模拟交易。
- 策略监控：低价擒牛、净利增长等策略的持仓跟踪与卖出提醒。

### 1.5 基础设施能力

- 数据源：AkShare、Tushare、yfinance、TDX、问财、qstock 等。
- LLM：DeepSeek/OpenAI 兼容接口。
- 存储：多个 SQLite 数据库文件。
- 通知：邮件、Webhook、钉钉/飞书。
- 报告：PDF、Markdown、HTML。
- 调度：实时监测、新闻流量、板块策略、持仓分析等定时任务。

## 2. 当前结构问题

### 2.1 `app.py` 职责过重

`app.py` 当前承担了以下职责：

- Streamlit 页面配置、CSS 和顶层布局。
- 侧边栏导航与页面路由。
- 单股分析页面。
- 批量分析页面。
- 单股/批量分析业务流程编排。
- 历史记录列表、详情和加入监测逻辑。
- 环境配置页面。
- 股票信息、图表、智能体报告、最终决策展示组件。

这导致入口文件体积过大，任何页面或业务逻辑变更都可能影响主入口。

### 2.2 顶层文件过多

当前顶层存在大量按前缀组织的文件，例如：

- `news_flow_*`
- `sector_strategy_*`
- `longhubang_*`
- `smart_monitor_*`
- `main_force_*`
- `low_price_bull_*`
- `macro_cycle_*`
- `macro_analysis_*`

这些文件已经隐含了业务边界，但还没有形成 Python 包结构，依赖关系仍暴露在项目顶层。

### 2.3 UI 与业务逻辑耦合

多个 `*_ui.py` 和 `app.py` 中同时包含：

- Streamlit 组件绘制。
- 业务流程编排。
- 数据源调用。
- AI 调用。
- 数据库存储。
- 通知发送。

这会让业务逻辑难以测试，也让同一能力难以复用于定时任务、命令行或 API。

### 2.4 基础设施重复

不同业务域中重复存在以下模式：

- `*_data.py` 数据抓取。
- `*_agents.py` 智能体 prompt 和 LLM 调用。
- `*_engine.py` 综合编排。
- `*_db.py` SQLite 存储。
- `*_pdf.py` 报告生成。
- `*_scheduler.py` 定时任务。

这些模式适合抽出统一接口和基础工具。

### 2.5 存储分散

项目当前有多个 SQLite 数据库文件，例如：

- `stock_analysis.db`
- `stock_monitor.db`
- `smart_monitor.db`
- `news_flow.db`
- `sector_strategy.db`
- `longhubang.db`
- `portfolio_stocks.db`
- `main_force_batch.db`

短期可以保留多库，但连接、事务、迁移、JSON 序列化和路径管理应统一。

## 3. 改造目标

### 3.1 短期目标

- 保持现有功能和页面行为不变。
- 将顶层散文件按业务域迁入包目录。
- 瘦身 `app.py`，让它只负责应用启动和路由分发。
- 把单股/批量分析流程从 UI 中抽到服务层。
- 建立统一的目录约定和 import 规范。

### 3.2 中期目标

- 统一 LLM 客户端、数据源接口、数据库连接、报告接口、通知接口。
- 让每个业务域形成稳定结构：`ui.py`、`service.py/engine.py`、`data.py`、`agents.py`、`repository.py`、`report.py`、`scheduler.py`。
- 为核心业务服务补充单元测试和最小回归测试。

### 3.3 长期目标

- 支持 Streamlit UI、定时任务、命令行或 API 复用同一服务层。
- 支持新增策略或数据源时只增加独立包，不污染顶层入口。
- 支持数据库迁移和配置版本化。

## 7. 关键设计约束

### 7.1 UI 层约束

- UI 文件可以 import 服务层。
- UI 文件不应被服务层、数据层、repository 层 import。
- UI 文件可以使用 `st.session_state`，但 Session State key 应集中管理。
- UI 文件不直接操作 SQLite，必须通过 repository/service。

### 7.2 服务层约束

- 服务层不直接依赖 Streamlit。
- 服务层返回结构化结果，不直接渲染页面。
- 服务层负责业务编排：数据获取、智能体调用、结果整理、持久化。
- 服务层接收显式参数，不从 `st.session_state` 读取配置。

### 7.3 数据层约束

- 数据抓取模块只负责获取和标准化数据。
- 第三方库异常要转换为可解释的业务错误。
- 数据源 fallback 逻辑集中到 provider 或 data service，不散落在 UI。

### 7.4 智能体层约束

- 智能体模块负责 prompt、输入组装、LLM 调用和结果结构化。
- 不应直接依赖 Streamlit。
- 不应直接写数据库。
- 业务域内 prompt 可以保留在对应 `agents.py`，公共 prompt 工具放 `integrations/llm/prompts.py`。

### 7.5 Repository 约束

- 所有 SQLite 访问集中在 repository 或 infrastructure DB helper。
- 不在 UI 中直接写 SQL。
- JSON 序列化、时间字段、连接关闭、事务提交统一处理。
- 迁移期间可保留多个 `.db` 文件，不强制合库。

## 8. 推荐的兼容迁移策略

为了降低一次性改动风险，建议采用“新包 + 旧文件转发”的方式：

```python
# old_module.py
from src.aiagents_stock.features.xxx.yyy import *
```

或更推荐在正式安装包名前使用：

```python
from aiagents_stock.features.xxx.yyy import *
```

迁移期间允许旧路径短期存在，但每个 shim 文件顶部应标注：

```python
"""Compatibility shim. Prefer importing from aiagents_stock.features.xxx.yyy."""
```

清理条件：

- 所有内部 import 已替换为新路径。
- 文档和启动脚本已更新。
- 测试和手工验收通过。
- 再删除旧 shim。

## 9. 风险点与处理策略

### 9.1 Streamlit 状态丢失

风险：拆分页面后，`st.session_state` key 清理不一致，导致页面跳转异常。

处理：

- 建立 `app/state.py`，集中定义页面 key。
- 提供 `switch_page(page_name)`、`clear_pages()` 等方法。
- 不在各页面手写大量 `del st.session_state[key]`。

### 9.2 循环依赖

风险：当前存在 UI 模块反向 import `app.py` 的情况，迁移后可能形成循环。

处理：

- 禁止业务包 import `app/main.py`。
- 将共享展示组件迁到 `app/components.py` 或对应 feature 的 `ui_components.py`。
- 将业务函数迁到 service，UI 间不要互调。

### 9.3 数据库路径变化

风险：移动 repository 后默认数据库路径变化，导致读取不到旧数据。

处理：

- 所有数据库默认路径保持项目根目录下原文件名。
- 通过配置集中管理 DB 路径。
- 迁移后先不改数据库文件名和表结构。

### 9.4 外部数据源不稳定

风险：模块化期间误把异常吞掉，导致页面无数据但无提示。

处理：

- 数据层返回统一错误结构。
- UI 层展示数据源失败原因。
- fallback 发生时结果中带 `is_fallback` 或 `source` 字段。

### 9.5 LLM 调用命名误导

风险：项目已支持 OpenAI 兼容模型，但类名仍叫 `DeepSeekClient`。

处理：

- 新增 `OpenAICompatibleClient`。
- `DeepSeekClient` 作为兼容别名保留一段时间。
- 文档统一称为 OpenAI 兼容 LLM 客户端。

## 13. 不建议一次性做的事

以下事项建议延后，避免和模块化混在一起扩大风险：

- 不要同时更换 Streamlit 框架。
- 不要同时合并所有 SQLite 数据库。
- 不要同时重写所有 prompt。
- 不要同时调整交易执行逻辑。
- 不要同时大规模修改 UI 样式。
- 不要在没有回归保护前删除旧文件。
